# TimeForge App Diary

## 🏗️ Architecture: Vertical Slice Architecture (VSA)

TimeForge has evolved from a traditional N-Layer MVC architecture to a modern **Vertical Slice Architecture (VSA)** using the **REAPR** pattern. This approach focuses on features rather than layers, ensuring that all code related to a single functional unit (e.g., "Create Project") resides in one place.

### Key Frameworks:
- **FastEndpoints**: Used for building highly performant and clean APIs. It replaces standard MVC controllers with focused endpoints.
- **EF Core**: Persistent storage using Entity Framework Core (currently In-Memory for development).
- **Xunit v3 & Shouldly**: A robust testing suite ensuring 100% coverage per slice.

## ⚙️ How Components Work

### 1. Feature Slices
Each slice is self-contained in `TimeForge.Api/Features/{FeatureName}/{ActionName}/`. A typical slice consists of:
- **Request**: DTO for incoming data (e.g., `CreateProjectRequest`).
- **Endpoint**: Inherits from `Endpoint<TRequest, TResponse>`, orchestrating the logic.
- **Response**: DTO for outgoing data (e.g., `ProjectResponse`).

### 2. Soft Delete Mechanism
Implemented as a cross-cutting concern to prevent data loss.
- **Base Model**: Entities inherit from `BaseDeletableModel<string>`.
- **Interceptor**: The `SoftDeleteInterceptor` in `Database/Interceptors/` automatically catches `Deleted` state calls and converts them to `Modified` with an `IsDeleted` flag.
- **Global Filters**: (Planned/Active) Ensuring deleted items are hidden from results by default.

### 3. Identity & Global IDs
- **Identity**: Built on ASP.NET Identity with JWT Bearer tokens.
- **Global ID Strategy**: Managed in `TimeForgeDbContext.cs`, ensuring consistent ID generation (Client-side for Guids, DB identity for ints).
